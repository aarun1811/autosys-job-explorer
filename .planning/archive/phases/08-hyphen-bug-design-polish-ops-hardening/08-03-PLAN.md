<objective>
Close OPS-04: add a Linux CI gate that catches portability regressions in `ops/rectrace-ops.sh` and `ops/components.sh` on every push. Two deliverables — a GitHub Actions workflow and an ops-side smoke that does NOT require booting Java + Node.

Purpose: macOS-specific bash features regress easily (`mapfile`, `declare -A`, `sed -i ''`). A Linux CI job catches these the moment they land. The smoke avoids booting the real stack on the CI runner — that would need Oracle + Elasticsearch + pnpm, which is out of scope for an ops-portability gate. Instead we exercise the dispatcher, the registry parse, idempotent PID handling, and the readiness-probe path using a fake/stub component pointing at a tiny inline HTTP server (`python3 -m http.server` is on every ubuntu-latest runner).

Output: One workflow file + one smoke script. Workflow includes a `[NEEDS USER REVIEW]` comment block per D-8.11 documenting the Citi-CI swap path.
</objective>

  <action>Create `ops/ci-smoke.sh` modeled on `scripts/smoke-ssrm.sh` style. Header:

  ```bash
  #!/usr/bin/env bash
  # ops/ci-smoke.sh — Linux portability smoke for ops/rectrace-ops.sh + ops/components.sh.
  # Does NOT boot the real stack. Stands up a stub HTTP server and a stub component entry,
  # then exercises start/status/stop/restart/logs dispatch + readiness probe + idempotency.
  set -euo pipefail
  ```

  Behaviour:

  1. **Setup**: cd to `$(git rev-parse --show-toplevel)`. Pre-flight assertions:
     - `bash --version` printed (for CI log).
     - `shellcheck` is on PATH (echo version, fail if missing).
     - `python3` is on PATH (echo version).

  2. **shellcheck gate**: `shellcheck -x ops/rectrace-ops.sh ops/components.sh`. Fail loud on non-zero.

  3. **Static sanity**: `bash -n ops/rectrace-ops.sh` and `bash -n ops/components.sh` (parse-only).

  4. **Stub HTTP server**: `python3 -m http.server 7799 --bind 127.0.0.1 >/tmp/stub-http.log 2>&1 &`. Save its PID to `/tmp/stub-http.pid`. Trap EXIT to kill it. `wait_ready` for `http://127.0.0.1:7799/` to return 200 (max 5s).

  5. **Source the registry, append a stub entry**: source `ops/components.sh`; append one entry `stub|7799|run/stub.pid|logs/stub.log|http://127.0.0.1:7799/|-|sleep 600` to `REGISTRY` (re-export). Write the augmented registry to `/tmp/components-with-stub.sh` and set `RECTRACE_COMPONENTS_FILE=/tmp/components-with-stub.sh` so the main script picks it up — IF the script supports this hook. If not (08-02 v2 does not declare this hook), instead clone `ops/components.sh` to `/tmp/components-with-stub.sh`, append the stub line via `sed`, and run the dispatcher via `bash -c 'source /tmp/components-with-stub.sh; source <(sed "s|source .*components.sh|source /tmp/components-with-stub.sh|" ops/rectrace-ops.sh)'`. PREFER the env-hook path: add a one-line edit to `ops/rectrace-ops.sh` honoring `${RECTRACE_COMPONENTS_FILE:-$SCRIPT_DIR/components.sh}` and source THAT. Document this hook in `08-02-SUMMARY.md` (defer to the executor — they may either inline the patch into this plan's Task 1 or amend 08-02 and re-commit). Adopt the env-hook path explicitly here: include a sentinel in this task's `<verify>` that asserts the dispatcher sourced the override.

  6. **Dispatch assertions** (each fails the smoke on non-zero):
     - `bash ops/rectrace-ops.sh` (no args) exits non-zero and prints usage.
     - `bash ops/rectrace-ops.sh status all` lists `stub` (proves registry override loaded).
     - `RECTRACE_READY_TIMEOUT=5 RECTRACE_READY_INTERVAL=1 bash ops/rectrace-ops.sh start stub` exits 0 (readiness probe hits the stub HTTP server) and `run/stub.pid` exists.
     - `bash ops/rectrace-ops.sh start stub` (second invocation) exits 0 AND prints `already running` (idempotency).
     - `bash ops/rectrace-ops.sh status stub` reports `up (pid N)`.
     - `bash ops/rectrace-ops.sh stop stub` exits 0; `run/stub.pid` is gone.
     - `bash ops/rectrace-ops.sh logs all` exits non-zero (no multi-tail per OPS-02 explicit).
     - Unknown component: `bash ops/rectrace-ops.sh status bogus` exits non-zero AND prints `Unknown component`.

  7. **Cleanup**: trap EXIT kills the stub HTTP server, removes `/tmp/stub-http.pid`, `/tmp/components-with-stub.sh`, `run/stub.pid` (if present).

  8. **Result**: print `[PASS] ci-smoke: N/N assertions` or fail loud with the failed assertion index.

  Implementation constraints:
  - `set -euo pipefail`.
  - `shellcheck` clean.
  - bash 3.2 portable (no `mapfile`, no `[[ ... =~ ... ]]` if avoidable, no `declare -A`).
  - Make executable: `chmod +x ops/ci-smoke.sh`.

  **Important compatibility note for the executor:** if Plan 08-02's `ops/rectrace-ops.sh` did not declare the `RECTRACE_COMPONENTS_FILE` env hook, ADD it here as a tiny edit to `ops/rectrace-ops.sh`: change the line `source "$SCRIPT_DIR/components.sh"` to `source "${RECTRACE_COMPONENTS_FILE:-$SCRIPT_DIR/components.sh}"`.   <action>Per D-8.11: create `.github/workflows/ops-script.yml`. Single job, single runner. Top of file:

  ```yaml
  # .github/workflows/ops-script.yml
  # [NEEDS USER REVIEW] — placeholder GitHub Actions workflow for OPS-04 (Phase 8).
  # When Citi CI integration is known (Jenkins / GitLab CI / TeamCity / Citi-hosted GHA runner),
  # swap the `runs-on:` value or port this workflow's two `run:` steps into the equivalent
  # pipeline YAML. The two steps (shellcheck + ops/ci-smoke.sh) are CI-agnostic.

  name: ops-script
  on:
    push:
      paths:
        - 'ops/**'
        - '.github/workflows/ops-script.yml'
    pull_request:
      paths:
        - 'ops/**'
        - '.github/workflows/ops-script.yml'
    workflow_dispatch: {}

  jobs:
    portability:
      runs-on: ubuntu-latest
      timeout-minutes: 5
      steps:
        - name: Checkout
          uses: actions/checkout@v4

        - name: Install shellcheck
          run: |
            sudo apt-get update -qq
            sudo apt-get install -y --no-install-recommends shellcheck

        - name: Print toolchain
          run: |
            bash --version
            shellcheck --version
            python3 --version
            curl --version | head -1

        - name: shellcheck ops scripts
          run: shellcheck -x ops/rectrace-ops.sh ops/components.sh ops/ci-smoke.sh

        - name: ci-smoke (Linux portability + dispatch)
          run: bash ops/ci-smoke.sh
  ```

  Path-filter the workflow so it only runs when `ops/**` or the workflow itself changes — keeps CI cheap and the signal focused. Add `workflow_dispatch: {}` so a maintainer can trigger it manually for ad-hoc revalidation.
