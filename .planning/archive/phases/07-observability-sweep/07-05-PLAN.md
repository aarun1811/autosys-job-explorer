  <name>Task 1: Add maven-enforcer-plugin Micrometer pin to both POMs</name>
  <files>backend/rectrace/pom.xml, rectrace-tlm-stats/pom.xml</files>
  <action>
In BOTH `backend/rectrace/pom.xml` and `rectrace-tlm-stats/pom.xml`, add the maven-enforcer-plugin to `<build><plugins>` (alongside the existing spring-boot-maven-plugin). Use the exact template from RESEARCH.md §3.8:

```xml
<plugin>
  <groupId>org.apache.maven.plugins</groupId>
  <artifactId>maven-enforcer-plugin</artifactId>
  <version>3.6.2</version>
  <executions>
    <execution>
      <id>enforce-micrometer-pin</id>
      <phase>validate</phase>
      <goals><goal>enforce</goal></goals>
      <configuration>
        <rules>
          <!-- OBS-08 — fail build if any sub-dep tries to pull a Micrometer
               version disagreeing with the Boot BOM (Open Question 4: NOT
               requireUpperBoundDeps which forces upgrades). -->
          <dependencyConvergence/>
          <bannedDependencies>
            <excludes>
              <exclude>io.micrometer:*:[2.0.0,)</exclude>
            </excludes>
          </bannedDependencies>
        </rules>
      </configuration>
    </execution>
  </executions>
</plugin>
```

`<phase>validate</phase>` binds the execution to the earliest Maven lifecycle phase so the guard runs even on `mvn validate` / `mvn compile` (not just `mvn verify`). The Boot parent POM's default enforcer binding is to `validate` — adding our own execution under the same phase chains them; both run.

Per D-7.8 and Open Q4: use BOTH rules. `dependencyConvergence` (no parameters) asserts every transitive path agrees on one version. `bannedDependencies` excludes Micrometer >= 2.0.0 — the second-line defence if Boot's BOM bumps to 2.x before we re-evaluate.

If Boot's parent POM already declares a maven-enforcer-plugin block (Boot 3.x usually does), MERGE the executions rather than overwrite — add an `<executions>` child that supplements the inherited one. Verify via `mvn -pl backend/rectrace help:effective-pom | grep -A20 maven-enforcer-plugin` after the edit.

Per `<scope_reduction_prohibition>`: deliver BOTH rules in BOTH POMs in this task. Not "convergence-only for now" or "single module is enough" — D-7.8 says both rules and OBS-08 applies to both modules.

Sections (each a function for shellcheck cleanliness):
