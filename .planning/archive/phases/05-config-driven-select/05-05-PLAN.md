    Algorithm:
      1. `SqlTabConfigV4 tab = configService.getTab(tabKey).orElseThrow(() -> new IllegalArgumentException("Unknown SQL tab: " + tabKey));`
      2. Build the whitelist: `Set<String> whitelistedFields = tab.getColumns().stream().map(ColumnDefinition::getField).collect(toSet());`
      3. Build ORDER BY clause:
         - Iterate `req.getSortModel()`. For each entry: if `!whitelistedFields.contains(s.getColId())` throw `IllegalArgumentException("Sort column not allowed: " + s.getColId())`. If `!"asc".equalsIgnoreCase(s.getSort()) && !"desc".equalsIgnoreCase(s.getSort())` throw IllegalArgumentException.
         - Join entries: `"ORDER BY " + s.getColId() + " " + s.getSort().toUpperCase()` joined by `, `.
         - If empty, omit ORDER BY entirely (Oracle 12c+ OFFSET works without ORDER BY but yields unstable ordering — document with a `// TODO Phase 6: encourage configured queries to include a stable inner ORDER BY` comment).
      4. Build WHERE clause for the outer wrapper from `req.getFilterModel()`:
         - Iterate the map. For each entry: if `!whitelistedFields.contains(colName)` skip with a warn log (or throw; pick throw for strictness — Research Anti-Patterns says "reject unknown keys with 400"). Use throw.
         - Phase 5 supports two operators: `equals` and `contains`. For `equals`: append `" AND " + colName + " = ?"`. For `contains`: append `" AND UPPER(" + colName + ") LIKE UPPER(?)"` with value bound as `"%" + value + "%"`. All other operators throw `IllegalArgumentException("Unsupported filter operator: " + op)` (deferred to Phase 6).
         - Collect bind params in order.
      5. Build the wrapped SQL string:
         ```
         SELECT * FROM ( <tab.query> ) sub WHERE 1=1 <filter clause> <order by> OFFSET ? ROWS FETCH NEXT ? ROWS ONLY
         ```
         Append two more binds: `OFFSET = req.getStartRow()`, `FETCH NEXT = req.getEndRow() - req.getStartRow()`. NOTE the page-size is the SECOND `?` — Pitfall 4 belt-and-suspenders complement to `setMaxRows`.
      6. Execute via `readonlyJdbcTemplate.execute(PreparedStatementCreator, PreparedStatementCallback)`:
         - Creator: `conn -> { PreparedStatement ps = conn.prepareStatement(wrappedSql); ps.setQueryTimeout(queryTimeoutSeconds); ps.setFetchSize(fetchSize); ps.setMaxRows(maxRows); for (int i = 0; i < params.size(); i++) ps.setObject(i+1, params.get(i)); return ps; }`.
         - Callback: `ps -> { try (ResultSet rs = ps.executeQuery()) { ColumnMapRowMapper mapper = new ColumnMapRowMapper(); List<Map<String,Object>> rows = new ArrayList<>(); int n = 0; while (rs.next()) rows.add(mapper.mapRow(rs, n++)); return rows; } }` — try-with-resources on `ResultSet` per Pitfall 6.
      7. Lowercase column names in each returned row: build a fresh `LinkedHashMap` with `entry.getKey().toLowerCase()` (Pitfall 5; mirror `OracleServiceV4.normalizeColumnNames`).
      8. Build `SSRMResponseV4`: `rows` = the lowercased list; `lastRow` = `req.getStartRow() + rows.size()` if `rows.size() < (endRow - startRow)` else `-1` (AG-Grid SSRM convention: -1 means "more rows available"). Mirror `SearchServiceV4.computeLastRow` if it exists in OracleServiceV4 / SearchServiceV4.
      9. Log info "SQL tab [{}] executed: {} rows, sql={}" at DEBUG level (capture SQL for triage; no PII in tab metadata).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP client → controller path variable | tabKey from URL — must be validated against in-memory tab registry; never reaches SQL string-builder. |
| HTTP client → SSRMRequestV4 sortModel/filterModel | User-supplied column names — must be whitelisted against configured columns[].field BEFORE any string interpolation. |
| Wrapped SQL string → Oracle | Per-statement caps + read-only account; the wrapped query is built from validated parts + parameterized binds. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-13 | T (Tampering) | sortModel[i].colId carries SQL fragment (e.g., `DROP TABLE x`) | mitigate | `whitelistedFields.contains(colId)` check; throws IllegalArgumentException → controller returns 400. Negative live probe in acceptance_criteria asserts this end-to-end. Does NOT inherit OracleServiceV4.buildOrderByClause's known bug (CONCERNS.md, Phase 9). |
| T-05-14 | T (Tampering) | filterModel column name carries SQL fragment | mitigate | Same whitelist check. Filter values bound as `?` params — never concatenated. |
| T-05-15 | D (Denial of Service) | Huge page size (`endRow - startRow > 1_000_000`) crashes JDBC | mitigate | Controller validates `(endRow - startRow) <= 1000` and returns 400 otherwise. Inner setMaxRows(maxRows=10000) is belt-and-suspenders. |
| T-05-16 | I (Information Disclosure) | Unknown tabKey leaks internal info via stack trace | mitigate | Controller returns 400 with generic message "Unknown SQL tab: <key>" — no stack trace to client. Stack trace logged server-side. |
| T-05-17 | E (Elevation of Privilege) | rectrace_readonly account used for non-SELECT operations | mitigate | Defense in depth: validator rejects non-SELECT at boot (Plan 04); Hikari `setReadOnly(true)` hint (Plan 03); DB account has only `GRANT SELECT` (Plan 02). Three-layer wall. |
| T-05-18 | D (Denial of Service) | Long-running query exhausts HikariCP pool | mitigate | `ps.setQueryTimeout(30)` kills queries at 30s; try-with-resources on ResultSet + Spring's PreparedStatementCallback closes Connection. Plan 06 smoke includes a soak test variant in evidence. |
</threat_model>
