<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Config JSON on classpath → JSqlParser AST → boot decision | A malformed/hostile JSON could attempt to land a DML/DDL via the `query` field; parser rejection is the primary defense. |
| Validator visitor → query AST | A malformed CTE or set-operation could bypass the WHERE/FETCH check if the visitor doesn't recurse. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-05-09 | T (Tampering) | Config author writes INSERT/UPDATE/DELETE/DDL in query field | mitigate | `SqlShapeValidator.validate` rejects non-Select at parse time; `@PostConstruct` throws → Spring context init fails → boot fails. Negative live probe in acceptance_criteria proves this end-to-end. |
| T-05-10 | T (Tampering) | Config author writes `SELECT * FROM rectrace_core` (no bounds) | mitigate | Validator rejects when neither WHERE nor FETCH/LIMIT present. Plan 01's `SqlShapeValidatorTest.rejectsInvalidShapes` covers this via the unbounded-SELECT case. |
| T-05-11 | I (Information Disclosure) | Duplicate tab keys masking earlier definitions | mitigate | `init()` detects duplicate keys via `.distinct().count() != .size()` and throws IllegalStateException. |
| T-05-12 | D (Denial of Service) | A CTE with a 1000-deep recursive WITH chain crashes the parser at boot | accept | JSqlParser 5.3 handles recursion depth via internal limits; if a future attack vector emerges, add a depth cap. Documented as Phase-9 follow-up. |
</threat_model>
