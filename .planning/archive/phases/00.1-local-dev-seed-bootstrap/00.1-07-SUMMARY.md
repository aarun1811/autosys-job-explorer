### Discovered KNOWN GAPS handed to Phase 1

**1. [Phase 1 BOOT-08 work] Backend `DataSourceConfig.java` hardcodes the script-executor call**
- **Found during:** Task 3 smoke step 1 (the WARNING-4 strict smoke command did exactly what it was designed to do).
- **Root cause:** `backend/rectrace/src/main/java/com/citi/gru/rectrace/config/DataSourceConfig.java` lines 41-42:
  ```java
  ScriptExecutor scriptExecutor = new ScriptExecutor();
  String decryptedPassword = scriptExecutor.executeScript("/opt/rectify/control/scripts/get_password.sh",
          serviceName.toUpperCase(), dbschema.toUpperCase());
  ```
  This invocation IGNORES `datasource.password` from the local profile and tries to invoke the missing shell script, producing an empty/garbage password → Oracle returns `ORA-01005: Login denied due to invalid password`.
- **Impact:** Hibernate `JdbcEnvironmentInitiator` cannot read schema metadata; the JPA EntityManagerFactory still bootstraps with a null dialect-detected datasource; downstream JPA queries fail at runtime. ES-only endpoints (V4 search, suggest, ES probes) still work because they don't touch the primary datasource.
- **Plan 07 had ALREADY documented the parallel rectrace-tlm-stats version of this gap; it had NOT realized the backend module had the same pattern. Strict smoke surfaced it.**
- **Phase 1 BOOT-08 disposition:** Either (a) make `DataSourceConfig.java` use `@Value("${datasource.password:}")` and fall back to the script executor only when the property is empty, OR (b) refactor both modules to a common `PasswordResolver` bean that picks property-driven vs script-driven at runtime based on a single config switch.

**2. [Phase 1 BOOT-08 work] `rectrace-tlm-stats` DatabaseConfig.java triple script-executor invocation**
- **Found during:** Plan 07 authoring (NOT a new discovery — already in the plan's WARNING-4 rationale).
- **Root cause:** `DatabaseConfig.java` lines 80, 108, 190 all unconditionally call `scriptExecutor.executeScript(passwordScriptPath, ...)`.
- **Phase 1 BOOT-08 disposition:** Same fix pattern as Gap 1.

**3. [Phase 1 BOOT-08 work] Lombok 1.18.30 ↔ Java 25 compile incompatibility**
- **Found during:** Task 1 verification on the user's laptop (default `JAVA_HOME` = Java 25 via Homebrew).
- **Root cause:** Lombok 1.18.30 (bundled with Spring Boot 2.7.16's BOM) does not generate `@Builder` accessor code under javac 25 — `mvn compile` succeeds but the bytecode lacks the `builder()` method, producing `NoSuchMethodError` at runtime when V4 endpoints try to use `CategoryResultV4.builder()`.
- **Workaround used:** Compile and run with `JAVA_HOME=/opt/homebrew/opt/openjdk@21/libexec/openjdk.jdk/Contents/Home`. Phase 0-01 SUMMARY already documents this (maven.compiler.release=17 added in Phase 0).
- **Phase 1 BOOT-08 disposition:** Boot 2.7 → 3.3.x upgrade pulls in Lombok 1.18.32+ which supports Java 25; bumping JDK target to 21 in the parent POM completes the Phase 1 fix.
