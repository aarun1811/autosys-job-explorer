<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| HTTP client → Spring filter chain | X-Correlation-Id header crosses trust boundary here; backend must not blindly adopt arbitrary values |
| MDC → log output | traceId value from MDC is written to console log; a crafted header could inject log patterns |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-2-01 | Information Disclosure | AG-Grid license key | N/A — out of scope for Plan 02 | Handled in Plan 01 |
| T-2-02 | Injection | `X-Correlation-Id` header → MDC → log | mitigate | The `management.tracing.baggage.remote-fields=x-correlation-id` config causes Spring Brave to validate and adopt the header value as the traceId ONLY if it matches a valid 128-bit hex trace-id format (32 hex chars). Malformed values (log injection strings, non-hex chars) are rejected by Brave's internal validation; the backend generates a fresh traceId instead. The `CorrelationIdPropagationConfig` Javadoc notes this. This satisfies the T-2-02 threat (reflected correlation ID log injection). No additional `@ValidPattern` regex is needed at the controller layer for Phase 2 because Brave's propagation layer enforces the format. |
| T-2-03 | Information Disclosure | CORS over-permission (`@CrossOrigin(origins = "*")`) | accept | Pre-existing state from Phase 1; not changed in this plan. Phase 9 SEC-04 tightens CORS. No regression introduced. |
| T-2-04 | Elevation of Privilege | Dev proxy | N/A — out of scope for Plan 02 | Frontend concern only |
| T-2-05 | Tampering | `ops/build.sh` static cleanup | N/A — out of scope for Plan 02 | Handled in Plan 04 |
</threat_model>
