Constructor injection:

- `private final LoaderTicker ticker;`

Three endpoints:

1. `GET /jobs` — `public ResponseEntity<List<LoaderJobSummaryV4>> listJobs(@RequestHeader(...) String loginId)`:
   - Log loginId per existing convention.
   - For each `LoaderJobDefV4 def` from `loaderConfig.getJobs()`:
     - Build a `LoaderJobSummaryV4`.
     - Set `key`, `alias = def.getTarget().getAlias()`, `schedule`, `timezone`.
     - Call `runHistory.lastN(def.getKey(), 1)`; if non-empty, set `lastRun = list.get(0)`.
   - Return 200 + the list. Never throws (empty list is a valid response).

2. `POST /jobs/{key}/run-now` — `public ResponseEntity<?> runNow(@PathVariable String key, @RequestHeader(...) String loginId)`:
   - Log: "User [{}] requested run-now for loader job [{}]".
   - `Optional<LoaderJobDefV4> defOpt = loaderConfig.getJob(key)`. If empty: return 404 + `createErrorResponse("UNKNOWN_JOB", "Unknown loader job: " + key)`.
   - `TaskResult<Void> r = ticker.runNow(defOpt.get())`.
   - If `r.wasExecuted()`: return 200 + the latest run record `runHistory.lastN(key, 1).get(0)` (the just-completed SUCCESS or FAILED record).
   - If `!r.wasExecuted()`: return 409 + a `RunNowConflictResponseV4` with `currentRun = runHistory.lastN(key, 1)` first element if RUNNING-status, else null. The 409 path is D-6.14.
   - Wrap in try/catch (Throwable) — return 500 + `createErrorResponse("INTERNAL", "Run-now dispatch failed")` and log.error with stack. The job body itself catches its own throwables (Plan 04), but the lock provider can fail.

3. `GET /jobs/{key}/runs` — `public ResponseEntity<?> getRuns(@PathVariable String key, @RequestHeader(...) String loginId)`:
   - Log loginId + key.
   - Validate job existence via `loaderConfig.getJob(key).isPresent()`; 404 with UNKNOWN_JOB if not.
   - Return 200 + `runHistory.lastN(key, 20)`.

Private helper `createErrorResponse(String errorType, String message)` — verbatim copy from `SqlSearchControllerV4` (4-field Map: status, error_type, message, timestamp).

Javadoc references:
- Class-level: D-6.8 (endpoint list), D-6.13 (authZ deferred to Phase 9 — header is logged but not validated), D-6.14 (run-now 409 contract), 06-RESEARCH.md Pitfall L8 (documenting 409 not 202).
- Method-level: each endpoint cites its requirement ID (LOADER-08).

Script 1 — `scripts/smoke-loader-alias.sh` (LOADER-03 boot-fail). Asserts that booting the backend with a loader config referencing a non-existent ES alias fails fast.

Prerequisites: Docker stack up; apply.py applied; backend NOT running.

Logic:
1. Write a temporary loader config to `/tmp/loader-config-bad-alias.json` containing one job with `target.alias = "this_alias_does_not_exist_$$"` (random suffix avoids collisions on re-run). Same shape as the real loader-config-v4.json but minimal (one job, one PK column, the rectrace_core query).
2. Verify the alias really is absent: `curl -fsS -o /dev/null -w "%{http_code}" "http://localhost:9200/_alias/this_alias_does_not_exist_$$"` — must print `404`. Failure prints "FAIL: test alias unexpectedly exists" and exits.
3. Boot the backend with the bad config overriding the classpath default. Use the `local` profile. Run as a background process for at most 60 seconds; if it exits within 60s, capture exit code; if still alive at 60s, kill it and fail "boot did not fail fast as expected".
   - Command: `cd backend/rectrace && timeout 60 mvn -q spring-boot:run -Dspring-boot.run.profiles=local -Dspring-boot.run.arguments="--loader-config.location=file:/tmp/loader-config-bad-alias.json" >/tmp/loader-boot.log 2>&1` — capture both stdout and stderr to a log.
4. Capture the exit code of the `timeout` command into `BOOT_EXIT`. Assertions:
   - `BOOT_EXIT != 0` — the JVM exited non-zero (a successful boot keeps running, never exits — `timeout` would return 124 only if we hit the deadline). We want BOOT_EXIT to be the JVM's failure code (typically 1), NOT 124.
   - `grep -q "does not exist in Elasticsearch. Refusing to boot" /tmp/loader-boot.log` — the specific Pattern 5 message from LoaderConfigService.
   - `grep -q "this_alias_does_not_exist_$$" /tmp/loader-boot.log` — the bad alias name appears in the error.
   If any assertion fails, print "FAIL: <reason>" + dump `/tmp/loader-boot.log` tail, exit 1.
5. Cleanup `/tmp/loader-config-bad-alias.json` and `/tmp/loader-boot.log`.
6. Print "PASS: alias boot-fail smoke green".

Script 2 — `scripts/smoke-loader-sigterm.sh` (LOADER-09 mid-batch flush). Asserts that SIGTERM during a loader run flushes the in-flight queue and leaves no orphaned RUNNING rows.

Prerequisites: Docker stack up; apply.py applied (so `rectrace_core` has 5 rows and `rectrace_core_alias` is live).

Logic:
1. Pre-flight: backend must NOT be running. If `lsof -i:6088 | grep -q LISTEN`, fail "another process holds port 6088".
2. Boot backend in background with the local profile: `cd backend/rectrace && mvn -q spring-boot:run -Dspring-boot.run.profiles=local >/tmp/loader-sigterm.log 2>&1 &`. Capture PID. Wait up to 90s for `curl -fsS http://localhost:6088/rectrace/actuator/health` to return 200 (the actuator is on classpath per Phase 2 BOOT cleanup).
3. Trigger an immediate run via the admin endpoint to control timing:
   `curl -fsS -X POST "${BASE_URL}/api/v4/loader-admin/jobs/rectrace_core_loader/run-now" -o /dev/null` — fire-and-forget; the run completes in ~1-2s on the 5-row seed.
   For a more realistic mid-batch SIGTERM, increase the source row count first: `cd ../rectrace-local-dev && .venv/bin/python -c "import oracledb,os; from dotenv import load_dotenv; load_dotenv(); c=oracledb.connect(user='rectrace', password=os.environ['RECTRACE_PWD'], dsn=os.environ['ORACLE_DSN']); cur=c.cursor(); cur.execute(\"INSERT INTO rectrace_core SELECT * FROM rectrace_core\"); c.commit(); cur.execute(\"INSERT INTO rectrace_core SELECT * FROM rectrace_core\"); c.commit(); cur.execute(\"INSERT INTO rectrace_core SELECT * FROM rectrace_core\"); c.commit()"` — doubles thrice → 40 rows. Increase further if needed for a useful mid-batch window; capture the final count for the assertion.
   Capture `ROWS_AT_SIGNAL` = `SELECT COUNT(*) FROM rectrace_core`.
   Re-trigger run-now: `curl -fsS -X POST ".../run-now"` — in the background so the script doesn't block waiting for completion.
   `sleep 0.5` to let the run enter the streaming loop.
4. Send SIGTERM: `kill -TERM <BACKEND_PID>`.
5. Wait for clean exit: `wait $BACKEND_PID` (or `while kill -0 $BACKEND_PID 2>/dev/null; do sleep 1; done` with a hard 90s deadline). Capture wall-clock elapsed.
6. Post-assertions:
   - `grep -q "Loader shutting down" /tmp/loader-sigterm.log` — the registry's @PreDestroy banner fired.
   - `grep -q "Loader shutdown complete" /tmp/loader-sigterm.log` — flush completed.
   - Oracle: `SELECT COUNT(*) FROM loader_run_history WHERE job_key = 'rectrace_core_loader' AND status = 'RUNNING'` — must be 0. (Either the run completed and was marked SUCCESS/FAILED, or it never started.) Use the same `oracledb` Python invocation pattern as the seed step.
   - ES: `curl -fsS "http://localhost:9200/rectrace_core_alias/_count" | grep -o '"count":[0-9]*'` — must show count == ROWS_AT_SIGNAL (or, if the run flushed partially before SIGTERM, count > 0 AND count is consistent with a clean batch boundary, never less than the count BEFORE the run).
7. Cleanup: revert the row-multiplication by re-applying the seed (`cd ../rectrace-local-dev && .venv/bin/python apply.py --reset >/dev/null`).
8. Print "PASS: sigterm flush smoke green".

If any assertion fails, dump `/tmp/loader-sigterm.log` tail.
