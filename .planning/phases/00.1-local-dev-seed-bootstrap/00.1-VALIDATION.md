# Phase 00.1 — Validation Strategy

> Per-phase validation contract for feedback sampling during execution.
> Phase 00.1 is unusual: the deliverable is dev tooling in a sibling repo, not product code in this repo.
> Validation is **manual smoke against the existing Boot 2.7.16 codebase** per CONTEXT.md D-0.1.23 — the seed must run the EXISTING app end-to-end before Phase 1 starts modifying anything.

## Test Infrastructure

| Property | Value |
|----------|-------|
| **Framework** | None — Phase 00.1 ships dev tooling, not testable code. No pytest / jest / mvn test runner is added inside the sibling repo (per CONTEXT.md "Deferred Ideas: CI for the sibling repo"). |
| **Config file** | n/a — no test runner. The "test gate" is the 8-item D-0.1.23 smoke checklist below. |

| **Estimated runtime** | `apply.py --verify` ~5s. Full D-0.1.23 smoke ~15-20 min (manual UI clicks). Stack cold-start ~3 min (Oracle 23c). |

---

## Sampling Rate

- **After every task commit (within Phase 0.1 execution):** Run `python apply.py --verify` if the commit touched `schema/`, `data/`, `apply.py`, `docker-compose.yml`, or `es/`. Skip for README / `.env.example` / `.gitignore` edits.
- **After every plan wave merge:** Run `python apply.py --reset && python apply.py --verify` end-to-end. Confirms idempotency (LOCAL-DEV-05) and that nothing in the wave broke earlier waves' artifacts.
- **Before `/gsd-verify-work`:** All 8 D-0.1.23 smoke checklist items return PASS against the existing Boot 2.7.16 codebase.
- **Max feedback latency:** 5 seconds for the per-task verify; 3 minutes for the per-wave full reset.

## Manual-Only Verifications

The 8-item D-0.1.23 smoke checklist is intrinsically manual — it exercises the EXISTING Angular UI + Spring Boot 2.7.16 + Oracle JDBC + ES HLRC stack against the freshly-seeded local stack. Automated end-to-end browser tests would be more ceremony than the phase's value.

| Behavior | Requirement | Why Manual | Test Instructions |
|----------|-------------|------------|-------------------|
| App starts on `local` profile | LOCAL-DEV-06 | Requires running `mvn spring-boot:run`; no automated harness in scope | `mvn -pl backend/rectrace spring-boot:run -Dspring.profiles.active=local` (using a temporary `application-local.properties` user crafts inline for the verification — Phase 1 will own the committed version). Confirm log line `Started RectraceApplication in NN seconds` with no `ORA-*` or `ElasticsearchException`. |
| V4 keyword search returns seeded rows | LOCAL-DEV-04 / LOCAL-DEV-06 | Exercises full backend SSRM + frontend AG-Grid path | `curl -X POST http://localhost:6088/rectrace/api/v4/search/initial -H 'Content-Type: application/json' -H 'x-citiportal-loginid: testuser' -d '{"searchTerm":"SAMPLE","category":"jobName"}'` — returns rows from seed. Alternative: `npm start` in `frontend/rectrace/`, open `http://localhost:4200`, click "Job Name" tab, search seed value. |
| Suggestion endpoint works | LOCAL-DEV-06 | Exercises ES suggestion path | `curl 'http://localhost:6088/rectrace/api/search/suggest?term=SAM&field=job_name'` returns ≥ 1 suggestion. |
| Execution-order graph renders multi-node | LOCAL-DEV-04 / LOCAL-DEV-06 | Frontend Cytoscape graph; visual verification | In the Angular UI, click the execution-order button on any seed row. Modal opens; graph shows 2-4 nodes (per AUTOSYS_TLM_RECON_SEQUENCES seed chain); node colors reflect `ujo_job` status. |
| TLM-stats modal opens with charts | LOCAL-DEV-04 / LOCAL-DEV-06 | Requires `rectrace-tlm-stats` running + Angular UI | `mvn -pl rectrace-tlm-stats spring-boot:run` (also with a temporary `application-local.properties` for the recportal + reconmgmt datasources). In Angular UI, click set_id renderer; modal opens; charts populated from 7-14 days of `mr_csum_*` data. |
| QuickRec modal opens with data | LOCAL-DEV-04 / LOCAL-DEV-06 | Same | Click recon_id renderer; QuickRec modal opens; data from `quickrec_stats_table` + `recportal_manual_match_table` visible. |
| Hyphenated identifier search via `.keyword` returns the seed doc | LOCAL-DEV-04a (Phase 8 dry-run target) | Validates ES `.keyword` multi-field is wired correctly so Phase 8's hyphen-bug fix will be runnable here later | `curl 'http://localhost:9200/rectrace_core_index/_search' -H 'Content-Type: application/json' -d '{"query":{"term":{"set_id.keyword":"SET-ABC-123"}}}'` returns exactly 1 hit.
