## Phase Boundary

Produce `../rectrace-local-dev/` ‚Äî a **standalone git repository** living as a sibling directory to the rectrace project, outside this repo and outside everything that ships to Citi. The sibling repo provides everything the user needs to bring up a local Oracle + Elasticsearch stack on their macOS laptop, populate it with coherent sample data covering every Rectrace UI flow, and run `backend/rectrace` + `rectrace-tlm-stats` against it via the `local` Spring profile (defined in Phase 1's D-1.14).

**What the sibling repo contains:**

1. **`docker-compose.yml`** ‚Äî Oracle Free 23c + Elasticsearch 8.13.4 single-node, both bound to `localhost:1521` / `localhost:9200`, both no-auth or default-auth (specifics in D-0.1.6). One `docker compose up` brings the stack online.
2. **`schema/`** ‚Äî Hand-written Oracle DDL `.sql` files for the seven tables Rectrace touches:
   - `rectrace_core` (rectrace primary schema ‚Äî the single table all 13 V4 search categories read from)
   - `ujo_job` (Autosys schema ‚Äî live job status; queried by `JobStatusService`)
   - `AUTOSYS_TLM_RECON_SEQUENCES` (Autosys schema ‚Äî execution-order parent/child chains)
   - `AUTOSYS_ALL_JOBS_DATA` (Autosys schema ‚Äî job details for the side panel)
   - `recon_bank` (reconmgmt schema ‚Äî recon engine / set_id lookup)
   - `mr_csum_man_match_stats_hist`, `mr_csum_man_match_details`, `mr_csum_netting_hist` (reconmgmt schema ‚Äî TLM-stats modal queries)
   - `quickrec_stats_table`, `recportal_manual_match_table` (recportal schema ‚Äî QuickRec modal queries)
3. **`data/`** ‚Äî Seed `.sql` + `.json` files implementing **5 fully-connected scenarios** (one per typical UI exploration), each scenario consistent across all the tables above. At least 2 of the 5 scenarios use hyphenated job identifiers (e.g. `LOAD-ABC-123`) so Phase 8's hyphen-bug fix is dry-runnable on the laptop before any Citi deployment.
4. **`apply.py`** (or split `apply_oracle.py` + `apply_es.py` ‚Äî planner picks) ‚Äî Python script(s) using `oracledb` (thin mode ‚Äî no Instant Client required) and the `elasticsearch` Python client. Reads `schema/*.sql` ‚Üí applies DDL; reads `data/*.sql` ‚Üí applies INSERTs; reads `data/*-es.json` ‚Üí bulk-loads docs into `rectrace_core_index`. Idempotent: each apply drops and recreates schema before reloading.
5. **`requirements.txt`** ‚Äî `oracledb>=2.0`, `elasticsearch>=8.13,<9`, `python-dotenv` (planner picks exact pins). pip-installable into a virtualenv or pyenv environment.
6. **`.env.example`** ‚Äî Connection placeholders the user copies to `.env` (gitignored) before running `apply.py`. Defaults assume `localhost:1521` + `localhost:9200` per Phase 1 D-1.14.
7. **`reset.sh`** ‚Äî Tiny convenience wrapper: `docker compose down -v && docker compose up -d && sleep N && python apply.py`. Optional polish; planner may skip if `apply.py --reset` is cleaner.
8. **`README.md`** ‚Äî Install requirements (Docker Desktop, Python 3.12+, `pip install -r requirements.txt`), how to bring the stack up, how to seed, how to point `backend/rectrace` + `rectrace-tlm-stats` at it via `-Dspring.profiles.active=local`, how to reset.

**Out of scope for Phase 0.1:**
- Any artifact inside the rectrace repo itself ‚Äî Phase 1 D-1.15 forbids Docker / docker-compose / seed scripts / Testcontainers inside this repo. Phase 0.1's deliverables live exclusively in the sibling repo.
- Real Citi production data, real schemas extracted from Citi DBs, real credentials. All seed data is fabricated; types are inferred from rectrace's SQL grep + `search-config-v4.json`, not from any Citi source.
- Performance / load testing data volumes. 5 connected scenarios (~50-80 rows total) is the size; the goal is "every UI flow has live data," not benchmarking.
- TLM-instances data (TLM1..TLM9 dynamic datasources defined in `tlm-instances.json`). Those are placeholder hosts in the rectrace-tlm-stats config; their per-instance pages aren't on the Phase 1 smoke path.
- Oracle Instant Client installation steps ‚Äî `oracledb` thin mode means it's unnecessary. README mentions Instant Client only as an OPTIONAL aid for `sqlplus` ad-hoc inspection.
- Production-style seed lifecycle (Liquibase, Flyway, schema migrations) ‚Äî explicitly out of scope; this is fabricated dev data, not a migration story.
- Wiring sample data to Phase 5 (config-driven SELECT) or Phase 6 (ES Loader) features. Those phases extend the seed when their own scenarios demand it ‚Äî Phase 0.1 seeds only what's needed for Phases 1 + 2 + 3 (Boot upgrade smoke + React Foundation + React Search Vertical Slice).
- Sample data for Angular-only features that will be `drop`ped per the parity matrix (Phase 0 D-11). Only data exercising features marked `port` / `replace-content-with-recviz` / `replace-fully-with-recviz` is seeded.
- Pushing `../rectrace-local-dev/` to any remote ‚Äî `git init` lands during execution; whether to push to a private remote is the user's separate choice and not part of phase deliverables.

## Implementation Decisions

### Sibling folder shape

- **D-0.1.1:** `../rectrace-local-dev/` is its own **standalone git repository** (`git init` at the sibling folder's root). Independent history. User can optionally push to a private remote for fresh-laptop reproducibility ‚Äî out of phase scope. Sibling folder is NEVER referenced from this repo's git history (no submodule pointer, no `.gitignore` entry inside this repo).
- **D-0.1.2:** Sibling folder location is `../rectrace-local-dev/` (literal) ‚Äî a peer directory of `autosys-job-explorer/`. Planner creates it at execution time; phase commits to THIS repo only record planning artifacts in `.planning/phases/00.1-local-dev-seed-bootstrap/`. Executable artifacts (DDL, seeds, Python scripts, README) commit to the sibling repo separately.
- **D-0.1.3:** Sibling repo has no Citi-specific naming, no Citi branding, no Citi data ‚Äî it's a generic-looking dev-tooling repo. Safe to push to a personal public/private remote without disclosure concerns.

### Sample data scope and design

- **D-0.1.4:** Seed data is organized as **5 fully-connected scenarios**, each scenario being a fake Rectrace "job" (or recon process) that flows through every Rectrace UI surface end-to-end:
  - Each scenario has 1 row in `rectrace_core` (Oracle) + 1 doc in `rectrace_core_index` (ES) with all 36 fields populated (per `search-config-v4.json` field union).
  - Each scenario has 1 row in `ujo_job` for live status (planner picks status mix ‚Äî RUNNING / SUCCESS / FAILED ‚Äî across the 5).
  - Each scenario has a 2-4-node parent/child chain in `AUTOSYS_TLM_RECON_SEQUENCES` so the execution-order graph renders something multi-node (not a single isolated dot).
  - Each scenario has 1 row in `AUTOSYS_ALL_JOBS_DATA` for the side panel.
  - Each scenario has 1 matching `recon_bank` entry so the TLM-stats modal can pivot from it, plus 7-14 days of `mr_csum_man_match_stats_hist`, `mr_csum_man_match_details`, `mr_csum_netting_hist` history so the modal charts aren't empty.
  - Each scenario has 1-3 rows each in `quickrec_stats_table` and `recportal_manual_match_table` for the QuickRec modal.
- **D-0.1.5:** **At least 2 of the 5 scenarios use hyphenated identifiers** (e.g. `LOAD-ABC-123`, `RECON-XYZ-42`) across `job_name`, `set_id`, `recon_id`, and `recon_portal_id` fields. This gives Phase 8's hyphen-bug fix (BUG-01..03) a free dry-run environment on the laptop ‚Äî you can verify the `.keyword`-subfield design works before any Citi deployment. Hyphenated scenarios MUST include their canonical form in both the parent ES field AND its `.keyword` subfield variant (the field union pulled from `search-config-v4.json` has both for the affected fields).
- **D-0.1.6:** Total row count is roughly **50-80 rows across all tables** organized into 5 narratives. Not a performance test ‚Äî sufficient breadth to exercise every cell renderer, every modal, every tab, and the execution-order graph. Specific narrative content (what jobs, what recon engines, what calendars, what apps) is the planner's design decision; aim for realistic-looking but obviously-fake values (e.g. `SAMPLE_TRADE_RECON_001`).

### Schema authoring approach

- **D-0.1.7:** DDL is **hand-authored** from public artifacts inside the rectrace repo. No Citi-side schema dump is requested. Sources:
  - `backend/rectrace/src/main/resources/search-config-v4.json` ‚Äî authoritative for `rectrace_core` Oracle column list AND `rectrace_core_index` ES field mapping (36 distinct fields, including `.keyword` subfield variants).
  - `backend/rectrace/src/main/java/com/citi/gru/rectrace/service/JobStatusService.java` ‚Äî `ujo_job` columns (SELECT clause line 48+).
  - `backend/rectrace/src/main/java/com/citi/gru/rectrace/service/ExecutionOrderService.java` ‚Äî `AUTOSYS_TLM_RECON_SEQUENCES` and `AUTOSYS_ALL_JOBS_DATA` columns.
  - `rectrace-tlm-stats/src/main/java/com/citi/gru/rectrace/tlmstats/service/TlmStatsV2Service.java` ‚Äî `recon_bank`, `mr_csum_man_match_stats_hist`, `mr_csum_man_match_details`, `mr_csum_netting_hist` columns.
  - `rectrace-tlm-stats/src/main/java/com/citi/gru/rectrace/quickrec/service/QuickRecStatsService.java` ‚Äî `quickrec_stats_table`, `recportal_manual_match_table` columns.
- **D-0.1.8:** Type inference rules: `VARCHAR2(4000)` for names / identifiers / pattern strings (default), `DATE` for any column queried with date-arithmetic / GROUP BY date, `NUMBER(10)` for COUNT/SUM/aggregation columns, `CLOB` for `command` / `description` (Execution Order details). Tighten reactively if a query rejects the default at runtime.
- **D-0.1.9:** Schemas in the local Oracle: create three schema users ‚Äî `RECTRACE` (owns `rectrace_core`), `AUTOSYS` (owns the three AUTOSYS_* tables), `RECONMGMT` (owns the four reconmgmt tables), `RECPORTAL` (owns the two quickrec tables). Each gets the minimum grants the app needs. The DBA user is whatever `gvenzl/oracle-free`'s default SYSTEM password resolves to (see D-0.1.13). Cross-schema reads use schema-qualified table names exactly as the production SQL does.
- **D-0.1.10:** **ES index mapping** for `rectrace_core_index` declares every field's parent type from `search-config-v4.json`'s `searchFields` list (mostly `text` for parent, `keyword` for `.keyword` subfield variants). For the hyphenated fields (`file_name_pattern`, `job_name`, `box_name`, `recon`, `recon_id`, `recon_portal_id`, `tlm_instance`, `machine`, `set_id`, `load_file_name_pattern`, `run_calendar`, `exclude_calendar`), the explicit `.keyword` multi-field IS the contract ‚Äî Phase 8's fix expects to query through it.

### Application mechanism

- **D-0.1.11:** Python with `oracledb` (thin mode) + official `elasticsearch` Python client. No `sqlplus`. No Oracle Instant Client requirement. No bash sqlplus wrappers.
- **D-0.1.12:** Script structure: planner picks between (a) single `apply.py` with `--oracle` / `--es` / `--reset` flags, or (b) split `apply_oracle.py` + `apply_es.py`. Either is fine; either is idempotent (drop-if-exists before create); either reads from `schema/` and `data/` subdirs declaratively rather than hard-coding paths in the script.
- **D-0.1.13:** Connection config lives in `.env` (gitignored), copied from `.env.example` (committed). Defaults match Phase 1's `application-local.properties` (`localhost:1521`, `localhost:9200`). Oracle creds default to `system / <password from docker-compose env>`; ES has security disabled per D-0.1.16 so no creds needed.
- **D-0.1.14:** Optional `reset.sh` wrapper provides a one-command stack reset (`docker compose down -v && docker compose up -d && wait_for_readiness && python apply.py`). Planner skips if `apply.py --reset` covers the same UX.

### Docker compose stack

- **D-0.1.15:** `docker-compose.yml` in the sibling repo defines two services:
  - **`oracle`** ‚Äî image `gvenzl/oracle-free:23-slim` (community-built Oracle Free 23c, no Oracle login required to pull, ~700MB, supports `/opt/oracle/scripts/startup/` for init hooks). Env: `ORACLE_PASSWORD=<dev-only-password>`. Port mapping `1521:1521`. Volume for data persistence (named volume, not bind mount, so it survives container recreation but is wiped on `docker compose down -v`).
  - **`elasticsearch`** ‚Äî image `docker.elastic.co/elasticsearch/elasticsearch:8.13.4` (matches Phase 1 D-1.2's Boot 3.3.x BOM transitively). Env: `discovery.type=single-node`, `xpack.security.enabled=false`, `ES.JAVA_OPTS=-Xms512m -Xmx512m`. Port `9200:9200`. Volume for data persistence.
- **D-0.1.16:** ES security is **disabled** locally (`xpack.security.enabled=false`). Phase 1's local profile (`application-local.properties` per D-1.14) connects without credentials. Matches the dev-only `ElasticsearchDevConfiguration` profile pattern already in the repo. Production / UAT keep security on ‚Äî Phase 9 territory.
- **D-0.1.17:** Oracle's default DB after `gvenzl/oracle-free:23-slim` starts is `FREEPDB1` (the pluggable). The JDBC URLs in Phase 1's `application-local.properties` connect to `jdbc:oracle:thin:@localhost:1521/FREEPDB1`. `apply.py` connects via the same service name.
- **D-0.1.18:** Container healthcheck wait ‚Äî Oracle takes 60-120s to be ready on first boot. `reset.sh` (or `apply.py`) polls `oracledb.connect()` with backoff until it succeeds, with a 5-minute timeout, before applying schema. ES is ready in ~20s.

### Claude's Discretion (planner decides)

- **`apply.py` structure** ‚Äî single file with flags vs. split `apply_oracle.py` + `apply_es.py`. Either works; pick the cleaner one.
- **DDL file granularity** ‚Äî one `.sql` per table, or one per schema. Either; planner picks based on readability.
- **Scenario narrative theme** ‚Äî generic placeholders (`SAMPLE_JOB_001..005`) vs. themed (e.g. "5 cash-equities trade reconciliation jobs across 3 calendars"). Planner picks based on what's clearest for future-self readers. Themed feels more realistic and helps you remember what each scenario tests; generic is faster to author.
- **ES bulk-load file shape** ‚Äî one JSON-NDJSON file with all 5 scenarios' docs, or one file per scenario. Either; the `elasticsearch` Python client's `helpers.bulk` handles both shapes.
- **Oracle schema-user passwords** ‚Äî derive from `gvenzl/oracle-free`'s default SYSTEM env or set per-schema passwords. Doesn't matter for local; pick simple defaults and document in `.env.example`.
- **Reset semantics** ‚Äî `docker compose down -v` (volume wipe, slow Oracle restart) vs. `apply.py --reset` (drop schemas and re-INSERT, much faster, container stays up). Bias toward the faster path for daily-use ergonomics; `compose down -v` is the nuclear option in README.
- **README structure** ‚Äî single README.md is fine; sectioned for first-time setup, daily-use cycle, troubleshooting, optional Oracle Instant Client install.
- **Whether to seed Quartz tables / ShedLock tables** ‚Äî Phase 6 (ES Loader) will need these when its scheduler lands. Phase 0.1 does NOT seed them (out of scope per Phase 0.1 boundary); Phase 6's seed extensions happen at Phase 6 plan/execute time. Sibling repo grows over time.
- **Whether the sibling repo's first commit message follows GSD's commit-message conventions** ‚Äî the sibling repo isn't a GSD project, so conventions are loose. Bias toward readable messages but no enforcement.

## Existing Code Insights

### Reusable Assets
- `backend/rectrace/src/main/resources/search-config-v4.json` is the single source of truth for `rectrace_core` table columns and `rectrace_core_index` ES fields. Planner unions every `searchColumn`, every `elasticsearch.searchFields[]`, every `columns[].field` across all 13 categories ‚Äî that gives the column list for both Oracle DDL and ES mapping. Direct, no inference required.
- Existing native-SQL patterns in `OracleServiceV4`, `ExecutionOrderService`, `TlmStatsV2Service`, `QuickRecStatsService` ‚Äî every column name the seed needs is greppable from these files. No `@Entity` ambiguity.
- `tlm-instances.json` shows the per-instance DataSource shape but is NOT seeded (out of scope).
- The `rectrace-tlm-stats` `tlm-stats-modal` v2 endpoint contract is over HTTP ‚Äî local stack works as long as the rectrace-tlm-stats service can reach its `reconmgmt` + `recportal` datasources, which Phase 0.1 provides.

### Established Patterns
- **No JPA entities, no ORM, no schema migrations** ‚Äî everything is native SQL via JdbcTemplate / direct EntityManager native queries. So Phase 0.1's DDL is hand-written; there's no Hibernate to derive schema from.
- **Schema-qualified table names** in TLM-stats SQL (`reconmgmt.mr_csum_*`) ‚Äî the local Oracle must have a `RECONMGMT` schema user owning those tables, OR the rectrace-tlm-stats config's `recportal.datasource.db-schema` and equivalent properties must point at whichever local user does. Planner picks the simpler shape (separate schema users matching the production schema names).
- **Profile-based properties** ‚Äî already the pattern (`application.properties` / `application-prod.properties` / `application-uat.properties` / `application-test.properties` / soon `application-local.properties`).
