# Phase 0.1: Local Dev Seed Bootstrap - Research

**Researched:** 2026-05-12
**Domain:** Local Oracle + Elasticsearch dev stack (Docker Compose) + Python-driven DDL/seed application
**Confidence:** HIGH on stack mechanics; HIGH on column inventory (extracted from repo source); MEDIUM on Apple-Silicon edge cases (gvenzl multi-arch confirmed; ES JVM flag a known workaround on some chips)

## Summary

Phase 0.1's research load is narrow because all macro decisions (Oracle image, ES image, Python toolchain, hand-authored DDL, 5-scenario seed) are locked in CONTEXT.md. The research closes ten specific gaps the planner needs: image-startup-script paths, healthcheck syntax, schema-user creation mechanics, exact column inventories for the seven non-`rectrace_core` Oracle tables, an explicit ES mapping shape for `rectrace_core_index`, Docker volume semantics, scenario-design shape examples, the verification checklist for D-0.1.23, and macOS Apple-Silicon pitfalls for both images.

Key verified findings: `gvenzl/oracle-free:23-slim` ships multi-arch (linux/arm64 supported since 23.5; current `23-slim` is 23.26.x), uses `ORACLE_PASSWORD` (not `ORACLE_PWD`), supports `/container-entrypoint-initdb.d/` for first-boot init and `/container-entrypoint-startdb.d/` for every-start scripts (NOT `/opt/oracle/scripts/startup/` — CONTEXT.md's mention is incorrect), exposes a `healthcheck.sh` command on PATH, and exposes `FREEPDB1` as the default PDB. The image has a built-in `createAppUser APP_USER APP_USER_PASSWORD [TARGET_PDB]` helper plus an `APP_USER` env var, but for *multiple* schema users (RECTRACE, AUTOSYS, RECONMGMT, RECPORTAL) the cleaner path is a single `01-create-schemas.sql` in `/container-entrypoint-initdb.d/` that issues `CREATE USER ... IDENTIFIED BY ... QUOTA UNLIMITED ON USERS; GRANT CONNECT, RESOURCE TO ...;` per schema. `python-oracledb` 4.x (current as of May 2026) defaults to thin mode — no `init_oracle_client()` required, no DYLD_LIBRARY_PATH. `elasticsearch-py` 8.13.2 is the patch to pin for ES server 8.13.4 (client minor must equal server minor to access all features).

**Primary recommendation:** Plan a six-deliverable wave structure for the sibling repo: (1) `docker-compose.yml` + `.env.example` + `.gitignore`, (2) schema DDL per Oracle user (4 files), (3) ES mapping definition + bulk-load NDJSON, (4) the 5 scenario INSERT scripts, (5) `apply.py` with `--reset` flag and retry-with-backoff connection probe, (6) `README.md` + `requirements.txt`. The `apply.py` script handles schema-user creation post-`createAppUser`-doesn't-fit-our-shape; init-script handles only schema-user CREATE/GRANT so DDL stays in `apply.py`-managed `.sql` files that are easy to re-apply (D-0.1.12 idempotency).

| Reset semantics (fast path) | `apply.py --reset` (drop+recreate schemas, no docker restart) | `docker compose down -v` (nuclear, ~120s restart) | D-0.1.14 + Claude's Discretion bias toward the faster path |

### Pattern 1: `gvenzl/oracle-free` First-Boot Init for Multi-Schema Users

**What:** Mount a single SQL file into `/container-entrypoint-initdb.d/` that creates all four schema users with grants. Runs once on first container boot when the Oracle datafile volume is empty. On subsequent starts (named volume retains state), the init script is NOT re-run — `apply.py` handles schema-table-level recreation only.
**When to use:** Always — this is the canonical way to bootstrap schema users in this image.
**Example (`init/01-create-schema-users.sql`):**
```sql
-- Source: https://github.com/gvenzl/oci-oracle-free#startup-script-support
ALTER SESSION SET CONTAINER = FREEPDB1;

CREATE USER rectrace IDENTIFIED BY rectrace_pwd
  DEFAULT TABLESPACE USERS QUOTA UNLIMITED ON USERS;
GRANT CONNECT, RESOURCE TO rectrace;

CREATE USER autosys IDENTIFIED BY autosys_pwd
  DEFAULT TABLESPACE USERS QUOTA UNLIMITED ON USERS;
GRANT CONNECT, RESOURCE TO autosys;

CREATE USER reconmgmt IDENTIFIED BY reconmgmt_pwd
  DEFAULT TABLESPACE USERS QUOTA UNLIMITED ON USERS;
GRANT CONNECT, RESOURCE TO reconmgmt;

CREATE USER recportal IDENTIFIED BY recportal_pwd
  DEFAULT TABLESPACE USERS QUOTA UNLIMITED ON USERS;
GRANT CONNECT, RESOURCE TO recportal;
```
Mount in `docker-compose.yml`:
```yaml
services:
  oracle:
    volumes:
      - oracle-data:/opt/oracle/oradata
      - ./init/01-create-schema-users.sql:/container-entrypoint-initdb.d/01-create-schema-users.sql:ro
```

### Pattern 2: `apply.py` Retry-with-Backoff Connection Probe

**What:** Cold-boot Oracle takes 60-120s on first run; container can be "healthy" before the listener fully registers `FREEPDB1`. `apply.py` polls `oracledb.connect()` with exponential backoff.
**Example:**
```python
# Source: python-oracledb readthedocs error-handling guidance; idiomatic Python
import oracledb, time

def wait_for_oracle(dsn, user, password, max_wait_s=300):
    deadline = time.time() + max_wait_s
    attempt = 0
    while time.time() < deadline:
        try:
            conn = oracledb.connect(user=user, password=password, dsn=dsn)
            conn.close()
            return
        except oracledb.DatabaseError as e:
            attempt += 1
            delay = min(2 ** attempt, 15)
            print(f"Oracle not ready (attempt {attempt}): {e}; retry in {delay}s")
            time.sleep(delay)
    raise RuntimeError(f"Oracle not ready after {max_wait_s}s")
```

### Pattern 3: ES Bulk Load via `helpers.bulk` from NDJSON

**What:** Read a `.ndjson` file, parse each line as a JSON document, yield as action dicts to `helpers.bulk()`.
**Example:**
```python
# Source: elasticsearch-py 8.13 helpers docs
from elasticsearch import Elasticsearch
from elasticsearch.helpers import bulk
import json

def load_ndjson(es: Elasticsearch, index: str, ndjson_path: str):
    def gen():
        with open(ndjson_path) as f:
            for line in f:
                doc = json.loads(line)
                yield {"_index": index, "_id": doc.get("id") or doc["job_name"], "_source": doc}
    success, errors = bulk(es, gen(), raise_on_error=True)
    print(f"Indexed {success} docs into {index}")
```

### Pattern 4: Idempotent Drop+Recreate for Oracle Tables

**What:** Each `schema/*.sql` starts with `DROP TABLE ... CASCADE CONSTRAINTS` wrapped in `BEGIN ... EXCEPTION WHEN OTHERS THEN NULL; END;` so first run (table doesn't exist) doesn't fail. Then `CREATE TABLE`.
**Example:**
```sql
BEGIN
  EXECUTE IMMEDIATE 'DROP TABLE rectrace_core CASCADE CONSTRAINTS';
EXCEPTION WHEN OTHERS THEN
  IF SQLCODE != -942 THEN RAISE; END IF;  -- ORA-00942: table or view does not exist
END;
/

CREATE TABLE rectrace_core (
  file_name_pattern       VARCHAR2(4000),
  ok_file_name            VARCHAR2(4000),
  ...
);
```

### Anti-Patterns to Avoid

- **Putting CREATE TABLE in `/container-entrypoint-initdb.d/`:** It runs only on first boot when the volume is empty. If the user wants to re-apply schema changes during development, they'd have to `docker compose down -v` first (~120s rebuild). Keep CREATE-USER in init scripts; keep CREATE-TABLE in `apply.py`-managed files. `[VERIFIED: gvenzl README — initdb runs once]`
- **Using `/opt/oracle/scripts/startup/`:** CONTEXT.md mentions this but it's WRONG for `gvenzl/oracle-free`. The correct paths are `/container-entrypoint-initdb.d/` (first-boot) and `/container-entrypoint-startdb.d/` (every-start). `[VERIFIED: github.com/gvenzl/oci-oracle-free README]`
- **Using `ORACLE_PWD` env var:** That's the `container-registry.oracle.com` image's env. `gvenzl/oracle-free` uses `ORACLE_PASSWORD`. `[VERIFIED: gvenzl README]`
- **Calling `oracledb.init_oracle_client(...)`:** Forces thick mode even though `oracledb` 2.x+ defaults to thin. Don't call this unless explicitly switching to thick. `[VERIFIED: python-oracledb readthedocs]`
- **Bulk-indexing without explicit `_id`:** Without deterministic IDs, re-running `apply.py` creates duplicates. Use `_id = doc["job_name"]` or another stable key. `[CITED: elasticsearch-py helpers docs]`
- **`docker compose down -v` as the default reset:** Wipes Oracle datafile volume; rebuild is 60-120s. Use `apply.py --reset` (drop+recreate schemas only, ~5s) for daily iteration. `[ASSUMED — D-0.1.14 + Claude's Discretion]`

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Oracle bulk loading | Loops of `INSERT VALUES (...)` for many rows | `cursor.executemany(stmt, rows)` from `oracledb` | One round-trip per batch; orders of magnitude faster on cold-boot Oracle. `[CITED: python-oracledb readthedocs]` |
| ES bulk indexing | Loops of `client.index(...)` | `elasticsearch.helpers.bulk(client, gen())` | Single bulk API call with batching, error aggregation, optional streaming. `[CITED: elasticsearch-py helpers]` |
| `.env` parsing | Hand-parsing `KEY=VALUE` from a file | `python-dotenv` `load_dotenv()` | Quote handling, comments, expansion, gitignore-safe pattern. `[ASSUMED — industry standard]` |
| Container readiness | `time.sleep(120)` then connect | Retry-with-backoff loop (Pattern 2) | Boot time varies 60-180s on macOS Apple Silicon; fixed sleep is brittle. `[VERIFIED: gvenzl issue #53 — variable boot]` |
| Container healthchecks | Custom shell script invoking `sqlplus` | Built-in `healthcheck.sh` on PATH inside the gvenzl image | Already tested upstream. `[VERIFIED: gvenzl healthcheck.sh on GitHub]` |
| Connection-string parsing | Hand-building TNS strings | Easy-Connect (`localhost:1521/FREEPDB1`) | Native to `oracledb` 2.x+ thin mode. `[VERIFIED: python-oracledb readthedocs]` |

**Key insight:** This phase is dev tooling — every line not written is a line that doesn't drift. Lean on `helpers.bulk`, `executemany`, `healthcheck.sh`, and `python-dotenv` rather than rolling primitives.

## Common Pitfalls

### Pitfall 1: Apple Silicon Elasticsearch Crash on Newer Chips
**What goes wrong:** ES 8.13.4 on Docker Desktop on M-series Macs has been reported to crash at startup with JVM SIGILL in `org.elasticsearch.nativeaccess.lib.PosixCLibrary.geteuid` on some M-series chips. `[VERIFIED: elastic discuss forum + github issue 134034]`
**Why it happens:** ES JVM tries to use Scalable Vector Extension (SVE) instructions that aren't reliably supported on all Apple Silicon variants.
**How to avoid:** Add `-XX:UseSVE=0` to `ES_JAVA_OPTS`. The full env value becomes `ES_JAVA_OPTS=-Xms512m -Xmx512m -XX:UseSVE=0`.
**Warning signs:** `Native memory allocation (mmap) failed`, `SIGILL` in JVM crash log, ES container exits ~10s after start.

### Pitfall 2: `gvenzl` Init Scripts Only Run on First Boot
**What goes wrong:** Adding a new schema user to `init/01-create-schema-users.sql` after the volume is initialized does nothing — `gvenzl/oracle-free` only re-runs init scripts when the datafile directory is empty. `[VERIFIED: gvenzl README]`
**Why it happens:** The image checks for pre-existing datafiles before running init scripts.
**How to avoid:** When iterating on the schema-user list during development, either (a) `docker compose down -v` to wipe the volume and re-run init, or (b) add the user via `apply.py` post-boot as SYSTEM. Recommend (a) during initial development; document (b) as the daily-iteration path in README.
**Warning signs:** "ORA-01017: invalid username/password" when `apply.py` tries to connect as a newly-added user that wasn't in the init script.

### Pitfall 3: `.keyword` Subfield Must Be Explicit in the ES Mapping
**What goes wrong:** If you create the index without an explicit mapping, ES dynamic mapping creates `.keyword` subfields automatically for string fields under 256 chars — but the rectrace query path relies on the subfield existing at exact-match query time. Phase 8's hyphen-bug fix depends on the `.keyword` subfield being declared in the mapping, not dynamically inferred (which can break for long values).
**Why it happens:** Dynamic mapping's `ignore_above: 256` truncation policy can silently exclude longer hyphenated identifiers.
**How to avoid:** Declare each of the 12 hyphen-prone fields explicitly as `{"type": "text", "fields": {"keyword": {"type": "keyword", "ignore_above": 8192}}}`. See Code Examples §ES mapping below for the full list.
**Warning signs:** Phase 1 BOOT-09 smoke fails for hyphenated identifiers in the seed (`LOAD-ABC-123` returns 0 rows even though the doc was indexed).

### Pitfall 4: `executemany` with Mixed Row Sizes
**What goes wrong:** `cursor.executemany(stmt, rows)` requires all rows to have the same number of columns in the same order. Mixing variant rows (e.g., some with `command` populated, some NULL) fails silently.
**Why it happens:** `oracledb` reuses prepared-statement bindings.
**How to avoid:** Always provide a complete row tuple; use `None` for unset NULLable columns explicitly.
**Warning signs:** `DPY-2007: expected ... but got ...`, partial INSERTs visible in `SELECT COUNT(*)`.

### Pitfall 5: ES Client Version Drift Warnings
**What goes wrong:** Running `elasticsearch-py` 8.14+ against ES 8.13.4 emits compatibility warnings on every call (in logs, not stderr by default — easy to miss).
**Why it happens:** Client minor > server minor; client targets features the server doesn't have.
**How to avoid:** Pin `elasticsearch>=8.13,<8.14` in `requirements.txt`. The latest 8.13 patch is 8.13.2. `[VERIFIED: pypi]`
**Warning signs:** `UserWarning: The client noticed that the server is not Elasticsearch and we do not support this unknown product` or version-mismatch warnings.

### Pitfall 6: `oracledb` 4.x Default Connection Pool Behavior
**What goes wrong:** `oracledb` 4.x added connection-pool defaults that may issue extra session-establishment commands on cold-boot Oracle.
**Why it happens:** Pool warmup tries to allocate min-pool sessions at create time.
**How to avoid:** Use `oracledb.connect()` (single connection) for `apply.py`, not `oracledb.create_pool()`. The seed-loading workload is single-threaded; no pool needed.
**Warning signs:** `apply.py` hangs at the connection step for >30s even after `wait_for_oracle()` returned.

### Pitfall 7: `docker compose down` (without `-v`) Doesn't Wipe Data
**What goes wrong:** `docker compose down` removes containers but keeps named volumes. Next `up -d` reuses the same Oracle datafiles — schema users persist, but the user expected a fresh state.
**Why it happens:** Named volume persistence is intentional for `compose down` semantics.
**How to avoid:** Document in README: `docker compose down` = stop only; `docker compose down -v` = nuclear reset (~120s rebuild on next `up`). For schema-only resets, use `apply.py --reset`.
**Warning signs:** Unexpected old data appearing after a "reset"; init scripts not re-running.

### Pitfall 8: `executemany` Date Format Mismatch
**What goes wrong:** Passing date strings instead of `datetime.date` / `datetime.datetime` objects causes ORA-01861 ("literal does not match format string").
**Why it happens:** `oracledb` thin mode is stricter about implicit type coercion than `cx_Oracle` was.
**How to avoid:** In `data/*.sql` files, use Oracle `TO_DATE('2024-01-15', 'YYYY-MM-DD')` literals. In `apply.py` if generating dates programmatically, use Python `datetime` objects directly in the binds.
**Warning signs:** ORA-01861 during INSERT phase.

### `apply.py` Skeleton

```python
# Source: idiomatic Python; python-oracledb 4.x + elasticsearch 8.13 docs
import argparse, json, os, time
from pathlib import Path
import oracledb
from elasticsearch import Elasticsearch
from elasticsearch.helpers import bulk
from dotenv import load_dotenv

ROOT = Path(__file__).parent

SCHEMAS = ["rectrace", "autosys", "reconmgmt", "recportal"]

def wait_for_oracle(dsn, user, password, max_wait_s=300):
    deadline = time.time() + max_wait_s
    attempt = 0
    while time.time() < deadline:
        try:
            conn = oracledb.connect(user=user, password=password, dsn=dsn)
            conn.close()
            return
        except oracledb.DatabaseError as e:
            attempt += 1
            delay = min(2 ** attempt, 15)
            print(f"Oracle not ready (attempt {attempt}): {e}; retry in {delay}s")
            time.sleep(delay)
    raise RuntimeError(f"Oracle not ready after {max_wait_s}s")

def apply_sql_file(conn, path: Path):
    sql_text = path.read_text()
    cursor = conn.cursor()
    # Split on /\n  (PL/SQL block delimiter); use simple ; split for plain DDL
    for stmt in (s.strip() for s in sql_text.split("\n/\n") if s.strip()):
        for sub in (s.strip() for s in stmt.split(";\n") if s.strip()):
            print(f"  Executing: {sub[:80]}...")
            cursor.execute(sub)
    conn.commit()

def apply_oracle(reset: bool):
    dsn = os.environ["ORACLE_DSN"]    # e.g., "localhost:1521/FREEPDB1"
    wait_for_oracle(dsn, "system", os.environ["ORACLE_PASSWORD"])
    for schema in SCHEMAS:
        with oracledb.connect(user=schema, password=os.environ[f"{schema.upper()}_PWD"], dsn=dsn) as conn:
            schema_sql = ROOT / "schema" / f"{schema_file_num(schema)}-{schema}.sql"
            data_sql = ROOT / "data" / f"{schema_file_num(schema)}-{schema}-inserts.sql"
            print(f"[{schema}] applying schema: {schema_sql.name}")
            apply_sql_file(conn, schema_sql)
            print(f"[{schema}] applying data: {data_sql.name}")
            apply_sql_file(conn, data_sql)

def apply_elasticsearch():
    es = Elasticsearch(os.environ.get("ES_URL", "http://localhost:9200"))
    mapping = json.loads((ROOT / "es" / "rectrace_core_index.mapping.json").read_text())
    if es.indices.exists(index="rectrace_core_index"):
        es.indices.delete(index="rectrace_core_index")
    es.indices.create(index="rectrace_core_index", body=mapping)
    def gen():
        with open(ROOT / "data" / "05-elasticsearch-bulk.ndjson") as f:
            for line in f:
                doc = json.loads(line)
                yield {"_index": "rectrace_core_index", "_id": doc["job_name"], "_source": doc}
    success, _ = bulk(es, gen(), raise_on_error=True)
    print(f"Indexed {success} docs")

def main():
    load_dotenv()
    parser = argparse.ArgumentParser()
    parser.add_argument("--reset", action="store_true", help="Drop+recreate all data (default behavior is idempotent)")
    parser.add_argument("--oracle-only", action="store_true")
    parser.add_argument("--es-only", action="store_true")
    args = parser.parse_args()
    if not args.es_only:
        apply_oracle(reset=args.reset)
    if not args.oracle_only:
        apply_elasticsearch()
    print("Done. Stack ready.")

if __name__ == "__main__":
    main()
```

## Open Questions

1. **Should the Phase 8 hyphen-bug seed include a scenario where the OLD (broken) ES query returns 0 rows but the NEW (`.keyword`) query returns 1 row?**
   - What we know: D-0.1.5 says 2+ scenarios must have hyphenated identifiers; the `.keyword` subfield must be declared so the fix is dry-runnable.
   - What's unclear: Whether `apply.py` should also seed a "negative control" — an ES document where the hyphenated value is intentionally NOT indexed under `.keyword`, to demonstrate the bug pre-fix.
   - Recommendation: Defer to Phase 8 plan; Phase 0.1 just ensures the `.keyword` subfield is wired correctly per D-0.1.10. Phase 8 can author the negative-control scenario in its own seed extension.

2. **Should the init script for Oracle schema-user passwords use env-var substitution (so the user can override) or hardcode?**
   - What we know: D-0.1.13 says `.env` (gitignored) holds connection config; defaults match Phase 1's `application-local.properties`.
   - What's unclear: Whether `/container-entrypoint-initdb.d/01-create-schema-users.sql` can read env vars (it cannot — SQL files passed to `sqlplus` inside the container don't see Docker env vars without a wrapper `.sh` script).
   - Recommendation: Plan to hardcode the four schema-user passwords in the init `.sql` AND in `.env.example`. They are local-dev-only and never near Citi data. Document in README that customizing requires editing both files + `docker compose down -v` + restart.

3. **Should `apply.py` validate that the existing Boot 2.7.16 codebase can actually connect, OR is that purely a user-driven step?**
   - What we know: D-0.1.23 says the user runs the codebase against the seed for verification.
   - What's unclear: Whether `apply.py --verify` should also run a quick set of test queries to confirm the data is queryable as expected (e.g., "SELECT COUNT(*) FROM rectrace.rectrace_core" returns 5).
   - Recommendation: Add `apply.py --verify` as a small, optional integration helper. It runs ~6 Oracle SELECTs (one per table) + one ES query (`GET rectrace_core_index/_search?q=*`). Returns counts. Useful for both Phase 0.1 self-check and as a smoke pre-flight before user runs Spring Boot.
