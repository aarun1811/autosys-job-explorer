<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| developer workstation → public/private git remote | If the sibling repo is later pushed, anything tracked becomes recoverable from the remote forever. The `.gitignore` is the only thing stopping `.env` (which the developer will create with real local-DB passwords) from being pushed. |
| developer workstation → installed Python interpreter | pip pulls packages from PyPI; pinned ranges (>=4.0,<5 etc.) limit drift but don't prevent supply-chain risk. Acceptable for local-dev tooling. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-00.1-01 | Information Disclosure | `.gitignore` / `.env` workflow | mitigate | Write `.gitignore` containing exact line `.env` BEFORE creating any other file. Commit `.gitignore` in the very first commit. Acceptance criteria assert `grep -c '^\.env$' .gitignore == 1`. |
| T-00.1-02 | Information Disclosure | Sibling repo accidentally pushed to public remote with Citi-looking identifiers | mitigate | Per D-0.1.3, README explicitly states "no Citi terminology"; later plans (00.1-05, 00.1-07) author scenarios with generic identifiers (`SAMPLE_TRADE_RECON_001`) and call this out in README. This plan does not seed any data so the threat is latent here but mitigated by the convention being documented up front in the README "What this is" section. |
| T-00.1-03 | Tampering | PyPI supply chain (`oracledb`, `elasticsearch`, `python-dotenv`) | accept | Local-dev tooling only; never deployed to Citi. Version pins limit drift. Risk acceptable for dev-only path. |
</threat_model>
